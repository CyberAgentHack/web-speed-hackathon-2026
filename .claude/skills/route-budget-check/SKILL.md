# ルート別パフォーマンス予算チェックスキル

## 前提条件

- サーバーが起動していること（`http://localhost:3000` に接続可能）
- Chrome ブラウザが起動していること（chrome-devtools MCP 用）

## 予算基準

| 指標 | 良好 | 要注意 | 危険 |
|------|------|--------|------|
| JS 転送量（gzip 後） | < 300 KB | 300-500 KB | > 500 KB |
| HTML サイズ（gzip 後） | < 50 KB | 50-200 KB | > 200 KB |
| API request 数 | < 5 | 5-10 | > 10 |
| LCP（Lighthouse） | < 2.5s | 2.5-4.0s | > 4.0s |
| CLS | < 0.1 | 0.1-0.25 | > 0.25 |
| TBT | < 200ms | 200-600ms | > 600ms |

## ワークフロー

### Step 1: 対象ルートの確認

採点対象の 9 ページ:

| ページ | パス |
|--------|------|
| ホーム | `/` |
| 投稿詳細（テキスト） | `/posts/:postId` |
| 投稿詳細（写真） | `/posts/:postId` |
| 投稿詳細（動画） | `/posts/:postId` |
| 投稿詳細（音声） | `/posts/:postId` |
| DM一覧 | `/dm` |
| DM詳細 | `/dm/:conversationId` |
| 検索 | `/search` |
| 利用規約 | `/terms` |

### Step 2: ルート別のリソース計測

chrome-devtools MCP を使って各ルートのネットワークリクエストを分析:

```
1. navigate_page → 対象 URL
2. list_network_requests → 全リクエストを取得
3. 以下を集計:
   - JS ファイルの数と合計サイズ
   - API コール（/api/v1/*）の数
   - HTML サイズ
   - 画像/動画/音声リクエストの数とサイズ
   - 総転送量
```

### Step 3: LCP 要素の発見経路確認

SPA なので全ページで LCP は JS 実行後に描画される。

| 経路 | 速度 | 説明 |
|------|------|------|
| HTML parse 中 | 最速 | SSR が実装されていればこの経路 |
| Preload | 速い | `<link rel="preload">` で事前読み込み |
| JS 実行後 | 遅い | React のレンダリング後に初めて要素が描画される |
| API fetch 後 | 最遅 | API レスポンスを待ってから描画 |

## ページ別の特徴と注意点

| ページ | 特徴 | 潜在的な問題 |
|--------|------|-------------|
| ホーム | タイムライン、動画自動再生、音声波形 | メディアリクエスト多数 |
| 投稿詳細（動画） | 動画自動再生 | 動画ファイルが大きい |
| 投稿詳細（音声） | 波形表示 | Web Audio API 処理 |
| DM一覧 | WebSocket 接続 | リアルタイム通信のオーバーヘッド |
| DM詳細 | WebSocket + 既読管理 | リアルタイム通信 |
| 検索 | kuromoji + BM25 + 感情分析 | 辞書ロード (17 MB) |
| Crok | Web-LLM + SSE + Markdown | LLM モデルロード |
| 利用規約 | 源ノ明朝フォント | フォントサイズ |
