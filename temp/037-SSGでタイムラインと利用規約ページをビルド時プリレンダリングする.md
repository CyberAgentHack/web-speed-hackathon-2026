# 037-SSGでタイムラインと利用規約ページをビルド時プリレンダリングする

## 概要

現在のCSR SPAでは、ブラウザが空の `<div id="app"></div>` を受け取り、JSダウンロード・実行後にはじめてコンテンツが表示される。シードデータはデプロイごとに固定（faker.seed(123)）なので、認証不要なページはビルド時にHTMLを事前生成できる。

PoC として `/terms`（API呼び出しなし）と `/`（初期30件のposts取得）をSSG化し、FCP/LCPの改善効果を検証する。

## 期待効果

- FCP/LCP の大幅改善（JS実行を待たずにコンテンツ表示）
- Lighthouse スコア: FCP×10 + LCP×25 = 最大35点/ページ の改善ポテンシャル

## 方針

- Webpackビルド完了後にSSGスクリプトを実行
- ビルド時にAPIサーバーを一時起動してデータ取得
- `renderToString` でHTML生成 → `dist/` に配置
- クライアントは `hydrateRoot` でハイドレーション

## 主な技術課題

### 1. ハイドレーション一致
- AppContainer の `isLoadingActiveUser` 初期値を SSG 時は `false` にする
- `activeUser=null`（未ログイン）でSSG生成、クライアント初期レンダリングも同じ状態にする
- Timeline の初期データを `window.__SSG_DATA__` 経由で復元

### 2. React.lazy と SSR の非互換
- SSGスクリプトではlazy不使用、対象コンポーネントを直接import
- クライアント側の AppContainer は lazy のまま維持

### 3. useEffect でのデータフェッチ
- SSG時は useEffect 未実行。`useInfiniteFetch` に `initialData` オプションを追加
- `window.__SSG_DATA__.posts` を初期データとして注入

### 4. BrowserRouter vs StaticRouter
- SSGスクリプトでは `StaticRouter` を使用
- クライアント側は `BrowserRouter` のまま

### 5. 静的ファイル配信の共存
- SSG済みHTML と SPAフォールバックの共存が必要
- `connect-history-api-fallback` の rewrites でルーティング

## 検証方法

1. `pnpm run build` 成功、dist/ にSSG済みHTML存在
2. JS無効でもコンテンツ表示確認
3. コンソールに hydration mismatch 警告なし
4. 非SSGページ（/search, /dm 等）が従来通り動作
5. VRT パス
6. Lighthouse FCP/LCP 改善確認
